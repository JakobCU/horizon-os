# Security Scrutiny Checklist

> **Used by**: Compliance Reviewer, Coordinator Hub, IP & Legal Expert
> **Source**: Horizon Europe Model Grant Agreement Art. 13, CL3 Work Programme security provisions
> **Applies to**: All Cluster 3 proposals — security scrutiny is MANDATORY for this cluster

---

## Overview

All Cluster 3 proposals undergo **security scrutiny** by the EC's security scrutiny panel. This is separate from the scientific evaluation. A proposal can score 15/15 in evaluation and still be blocked by security scrutiny. This checklist ensures the proposal proactively addresses security concerns.

## Part A: Classification & Sensitivity

### A.1 EU Classified Information (EUCI)

- [ ] Does the project require access to EU Classified Information?
  - If YES: specify the classification level (RESTREINT UE/EU RESTRICTED, CONFIDENTIEL UE/EU CONFIDENTIAL, SECRET UE/EU SECRET)
  - If YES: identify which partners need Facility Security Clearance (FSC) and Personnel Security Clearance (PSC)
  - If NO: explicitly state this in the proposal

- [ ] Will the project PRODUCE classified information or results?
  - If YES: describe the Security Classification Guide (SCG) approach
  - If YES: identify the National Security Authority (NSA) for the coordinating country

### A.2 Sensitive Information (non-classified)

- [ ] Does the project handle sensitive but unclassified data? (law enforcement data, personal data, critical infrastructure details)
- [ ] Is there a plan for handling sensitive foreground (results)?
- [ ] Are there restrictions on publication of results?

## Part B: Dual-Use & Misuse Potential

### B.1 Dual-Use Assessment

- [ ] Could the project results be used for military purposes?
- [ ] Could the technology be exported and misused by hostile actors?
- [ ] Does the project involve items on the EU Dual-Use Regulation (2021/821) control list?
- [ ] Is there a credible misuse scenario that needs addressing?

### B.2 Misuse Mitigation

For each identified risk, describe:
1. The dual-use/misuse scenario
2. Mitigation measures (access controls, design constraints, ethical guidelines)
3. Residual risk assessment

## Part C: Third-Country Participation

### C.1 Non-EU/Non-Associated Countries

- [ ] Are there any partners from non-EU, non-Associated Countries?
  - If YES: is their participation essential and justified?
  - If YES: will they have access to sensitive information or technology?
  - If YES: are there export control implications?

### C.2 Entity Checks

- [ ] Have all partners been checked against EU sanctions lists?
- [ ] Are any partners owned or controlled by third-country entities?
- [ ] Are there any foreign direct investment (FDI) screening concerns?

## Part D: Ethics & Fundamental Rights

### D.1 Surveillance & Privacy

- [ ] Does the project develop surveillance capabilities?
- [ ] Is there a fundamental rights impact assessment?
- [ ] Does the project comply with GDPR and the Law Enforcement Directive (LED)?
- [ ] Are there safeguards against mass surveillance or disproportionate monitoring?

### D.2 AI & Automated Decision-Making

- [ ] Does the project use AI for security/law enforcement decisions?
- [ ] Is there human-in-the-loop oversight?
- [ ] Does the AI system fall under the EU AI Act's "high-risk" category?
- [ ] Are explainability and bias mitigation measures in place?

### D.3 Vulnerable Groups

- [ ] Could the project disproportionately affect vulnerable groups (migrants, minorities, children)?
- [ ] Are there safeguards to prevent discriminatory outcomes?

## Part E: Security Advisory Board

For CL3 projects, a Security Advisory Board (SAB) is often expected:

- [ ] Is an SAB planned?
- [ ] Does the SAB include national security authorities or designated representatives?
- [ ] Is the SAB's role in reviewing sensitive deliverables defined?
- [ ] Is the process for security-cleared deliverable review described?

## Part F: Proposal Text Requirements

The following should appear in the proposal (typically in Section 3.2 Management or a dedicated Security section):

### Minimum Required Statements

1. **Classification statement**: "This project [does/does not] require access to EU Classified Information at the level of [X]."

2. **Security handling plan**: "Sensitive information will be managed through [describe measures: secure communication channels, access controls, data handling protocols]."

3. **Dual-use statement**: "The consortium has assessed the dual-use potential of the expected results. [Assessment summary]. Mitigation measures include [list]."

4. **Third-country statement** (if applicable): "The participation of [partner from third country] is justified by [reason]. They will [not] have access to security-sensitive information. [Describe access restrictions if applicable]."
